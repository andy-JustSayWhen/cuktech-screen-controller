# 固件制作与刷机安全规则

本文适用于任何固件制作、修改、上传、下载校验、安装、回滚或恢复操作，不限于实时加载器、页面或菜单修改。

## 1. 先区分操作风险

- 仅离线制作和分析固件不等于向设备刷机；不得把“构建成功”描述成“设备已更新”。
- 仅上传固件或让设备只下载校验，不得切换启动分区或安装。
- 任何会写入固件分区、资源分区、启动分区、启动引导程序或分区表的动作都属于刷机，必须执行本文全部门槛。
- 能通过图片、配置、临时内存或本机服务实现的需求，不得通过刷机实现。

## 2. 制作前必须建立恢复基线

- 唯一确认目标设备的型号、当前固件版本和所属账号；型号或版本不一致立即停止。
- 保存一份未经修改的匹配版本官方固件，并记录来源、版本、字节数和官方文件指纹；原版固件不得被构建脚本覆盖。
- 在修改前确认设备仍在线、米家账号仍可控制设备、更新状态仍可查询，并准备好使用原版固件执行“只下载校验”和“正式安装”的命令。
- 明确当前可用的恢复层级：云端无线恢复、厂家恢复模式、拆机调试接口。未经验证不得声称设备具有自动回滚、双分区保护、按键恢复或可用的拆机刷写接口。
- 如果修改可能影响启动、分区、网络、米家后台服务或固件更新任务，而又没有经过验证的硬件恢复通道，不得安装到设备。

## 3. 固件修改必须最小化

- 不猜测地址、偏移、调用约定、内存布局或分区布局；所有修改位置必须对当前精确版本重新验证。
- 默认禁止修改启动引导程序、恢复区、分区表、无线网络初始化、米家通信和固件更新链路。确需修改时，必须单独取得用户明确授权，并先验证可用的硬件恢复方法。
- 每一版实验固件只引入一个可独立验证的最小变化，不把多个未知修改合并后一次安装。
- 优先保留并调用原厂逻辑，再附加自定义逻辑；自定义数据缺失、校验失败或执行异常时，应回到原厂行为。
- 不得把已经注入载荷的固件再次交给会覆盖相同资源区或载荷区的工具处理。
- 为每个输出固件生成不可混淆的文件名和清单，记录输入文件指纹、输出文件指纹、修改区间、载荷大小、目标地址和构建参数。

## 4. 安装前必须完成离线验证

- 校验文件头、精确字节数、尾部长度、全部校验值和输出文件指纹。
- 回读并逐字节确认所有修改区间；确认修改范围之外没有意外差异。
- 验证挂接目标、跳转目标、调用约定、载荷边界、资源区边界和重定位结果；任何一项不确定都不得安装。
- 运行对应自动测试，并保留构建清单、差异摘要、回读结果和测试结果。
- 从“只下载校验”到“正式安装”必须使用同一个已经校验的成品文件；期间不得重新构建、重新打包或原地覆盖。

## 5. 正式刷机必须经过即时确认门

- 先让设备只下载并校验成品固件，确认不会安装、不会切换启动分区。
- 在真正安装前，向用户展示目标设备、当前版本、目标文件、文件指纹、修改范围、已完成验证、主要风险和原版恢复方案，并取得针对这一次安装的明确确认。
- 早先的概括授权、离线期间的授权或对其他固件的授权不能代替本次即时确认。
- 安装期间保持设备稳定供电和联网，不拔电、不复位、不切换网络，不并行下发其他设备命令。
- 超时、短暂离线或状态未知时不得盲目重复安装；先查询更新状态、运行时间、云端错误记录和设备在线状态。

## 6. 安装后的成功标准

- 必须分别确认设备接受更新、下载完成、安装阶段出现、设备重启、运行时间重新计数并重新上线。
- 必须验证原厂基础功能、网络与米家通信、固件更新能力以及本次目标功能；只看到屏幕亮起不能视为成功。
- 设备仍在线但页面异常时，优先使用未经修改的原版固件通过云端无线恢复。
- 设备无法联网时立即停止反复刷机和无证据的连续断电重启，保存最后状态、日志、成品固件及其文件指纹，再判断厂家恢复模式或硬件恢复。
- 拆机、短接触点、连接调试器或直接写存储芯片属于新的高风险操作，必须先识别芯片和接口、制定接线与备份方案，并再次取得用户明确授权。

## 7. 强制停止条件

- 设备身份、固件版本、原版固件来源或文件指纹无法确认。
- 原版恢复文件或恢复命令没有准备好。
- 修改触及启动或联网关键路径，但没有已验证的恢复通道。
- 构建产物在验证后发生变化，或上传、下载和安装所用文件无法证明逐字节一致。
- 下载校验失败、设备报告更新错误、供电或网络不稳定、设备状态无法判断。
- 任何人要求跳过校验、即时确认门或安装后验证。
