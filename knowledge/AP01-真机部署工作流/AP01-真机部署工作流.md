# AP01 真机部署工作流

## 1. 目的与适用范围

本工作流用于把一台原厂酷态科 AP01 配置为可持续显示本机生成内容的局域网信息屏。它覆盖一次性实时加载器安装、任意图片验证、动态数据面板、登录自动启动和真机验收。

当前已经验证的固件修改范围仅包括：

| 项目 | 已验证值 |
| --- | --- |
| 设备型号 | `njcuk.enstor.ap01` |
| 原始固件 | `1.0.2_0031` |
| 设备画面 | 320×240、GIF89a、至少两帧 |
| 日常刷新位置 | `/tmp/.ap01q0.gif`、`/tmp/.ap01q1.gif`、`/tmp/.ap01q2.gif` |
| 默认刷新周期 | 300 秒 |

其他型号或固件版本必须重新定位、构建和验证固件修改位置，不能复用本工作流中的固件产物。

实测案例见：[2026-07-23：原厂 AP01 部署实时加载器与 Codex 面板](cases/2026-07-23-原厂AP01部署实时加载器与Codex面板.md)。

## 2. 核心结论

日常图片并不是由电脑主动写入设备。电脑只提供最新的 `/screen.gif`，AP01 主动下载、校验并写入三个轮换的临时内存槽位。只有首次安装实时加载器时会改写固件存储。

```mermaid
flowchart LR
    A["本机渲染图片或数据面板"] --> B["局域网画面服务<br/>/screen.gif"]
    B -->|"AP01 定时下载"| C["格式与体积校验"]
    C --> D["三个临时内存槽位轮换"]
    D --> E["虚拟形象页面更新"]
```

远程 OTA（通过网络安装固件）使用另一条链路：

```mermaid
flowchart LR
    A["构建并校验 AP01 固件"] --> B["小米云文件服务"]
    B --> C["官方固件分发地址"]
    C --> D["小米云向目标 AP01 下发安装指令"]
    D --> E["AP01 下载、校验、安装、重启"]
```

## 3. 依赖与准备

开始前先读取项目根目录的 `Goal.md`、`Depends.md` 和 `AGENTS.md`，并满足下列条件：

| 依赖 | 要求 | 验证方式 |
| --- | --- | --- |
| 目标设备 | AP01 已通电、加入米家并在线 | 小米云设备列表与设备截图交叉确认 |
| 型号与固件 | 必须精确匹配已验证范围 | 只读查询型号、固件版本和在线状态 |
| 本机环境 | Python 隔离环境及项目依赖完整 | 运行平台诊断脚本 |
| 小米登录 | 可复用的二维码登录状态 | 凭据只写入已忽略的 `.env` |
| 局域网 | AP01 与画面服务之间无终端隔离 | 双向只读探测与实际取图日志 |
| 本机地址 | 地址稳定，屏幕服务端口可访问 | 路由器只为目标电脑保留地址 |
| 供电与网络 | 安装期间持续稳定 | 不拔电、不复位、不切换网络 |

任何密码、登录令牌、设备内部编号、临时固件地址和固件文件都不得写入知识库或提交到仓库。

## 4. 标准部署流程

### 阶段 A：只读识别

1. 运行平台诊断脚本。
2. 使用米家登录状态查询目标 AP01。
3. 交叉确认型号、固件版本、局域网地址、硬件地址和在线状态。
4. 检查日志中是否已经存在目标设备读取 `/screen.gif` 的成功记录。
5. 如果设备已经具备实时加载器，跳过所有固件安装步骤。

完成标准：目标设备唯一，型号和固件无冲突，是否已安装加载器有明确证据。

### 阶段 B：建立稳定网络

1. 确认 AP01 与电脑处在同一未隔离局域网。
2. 在路由器中仅为画面服务电脑保留固定地址。
3. 启动画面服务并检查本机健康地址和 `/screen.gif`。
4. 保留 AP01 到电脑的实际访问日志。

路由器批量绑定按钮可能同时修改多个设备。任何批量操作后都要重新读取绑定列表，只保留目标电脑的预期变更。

### 阶段 C：取得并验证原厂固件

1. 通过小米云读取目标型号的官方固件信息。
2. 下载官方固件。
3. 核对文件头、文件大小和官方文件指纹。
4. 保存原始固件和构建产物到已忽略的 `artifacts/`，不得提交。

完成标准：原始文件逐字节完整，型号和版本精确匹配。

### 阶段 D：构建实时加载器

1. 先把一张有效的 320×240 动图写入原厂虚拟形象资源，作为离线兜底画面。
2. 编译设备端加载器。
3. 把加载器注入已验证的固件空闲区域。
4. 修改网络下载、画面更新和失败退避调用位置。
5. 写入电脑的稳定局域网画面地址。
6. 重新计算固件长度、恢复区校验值和构建报告。
7. 核对修改范围、程序符号、函数地址、文件头和最终文件指纹。

兜底画面只在实时画面尚未挂载或页面重新创建时出现。最终日常内容仍来自电脑提供的 `/screen.gif`。

### 阶段 E：验证云端下载链路

AP01 本身不能申请小米云文件上传地址。上传端必须使用当前账号中真正具备文件服务能力的小米网关；目标 AP01 只用于接收后续安装指令。

1. 上传已经构建完成的固件，取得短时有效的官方分发地址。
2. 回读分发地址的固件文件头。
3. 先执行“仅下载”验证。
4. 确认 AP01 下载完成、文件指纹匹配且没有切换启动分区。
5. 上传端和安装端必须使用逐字节相同的固件文件。

### 阶段 F：一次性安装确认门

真正安装前，必须向用户展示：

- 目标型号和固件；
- 原始与修改后文件指纹；
- 文件大小与文件头；
- 固件修改范围和校验结果；
- “仅下载”验证结果；
- 安装会写入固件存储并导致设备重启。

只有用户在此时明确授权，才允许发送安装指令。

### 阶段 G：安装与重启验收

安装过程中持续观察：

1. AP01 接受安装指令；
2. 下载进度到达 100%；
3. 状态进入安装阶段；
4. 设备短暂离线；
5. 设备运行时间重置；
6. AP01 重新在线。

仅看到进程退出、进度字段变化或设备短暂离线，都不能单独证明安装成功。

### 阶段 H：先验证任意图片

1. 将普通图片转换为 320×240 动图。
2. 验证 GIF89a、帧数、体积和完整结尾。
3. 切换为自定义图片模式并重启画面服务。
4. 等待 AP01 主动读取。
5. 日志必须出现目标设备的 `/screen.gif` 成功请求。

这一步用于证明完整链路，而不是只证明电脑能打开图片地址。

### 阶段 I：部署动态数据面板

1. 从本机官方登录状态读取额度和消耗数据。
2. 明确每个字段的统计口径和单位。
3. 使用设计效果图确定布局，再生成无文字底图。
4. 由本机程序绘制真实文字、圆环、折线和柱状图。
5. 对缺失字段显示“暂无数据”，不得编造。
6. 生成 320×240、至少两帧且低于体积上限的设备文件。
7. 切换到最终面板模式。

### 阶段 J：登录自动启动与持续验收

1. 安装当前用户登录自动启动服务。
2. 验证健康检查、画面文件和服务进程。
3. 观察至少两轮 300 秒自动刷新。
4. 第二轮刷新后，再确认 AP01 成功读取最新画面。
5. 确认最终驻留模式正确。

## 5. 验收证据总表

| 证据 | 通过条件 |
| --- | --- |
| 设备身份 | 型号、固件、在线状态和局域网目标一致 |
| 固件构建 | 文件头、长度、校验值、修改范围和文件指纹全部通过 |
| 云端下载 | AP01 完成下载和文件指纹校验，未安装测试不切换启动分区 |
| 固件安装 | 出现安装阶段、设备重启、运行时间重置和重新上线 |
| 图片文件 | 320×240、GIF89a、至少两帧、体积未超限 |
| 本机服务 | 健康检查正常、画面地址可读、最终模式正确 |
| 真机取图 | 目标 AP01 的 `/screen.gif` 请求返回成功 |
| 自动刷新 | 两轮连续刷新成功，第二轮后设备仍成功取图 |
| 自动启动 | 当前用户重新登录后服务能自动运行 |

## 6. 已知行为与改进方向

### 页面重新创建时短暂显示兜底图

物理按钮或旋钮可能让虚拟形象页面重新创建。当前加载器只记录“这一代画面是否应用过”，没有记录画面控件是否被重新创建，所以新页面可能先显示固件内置兜底图，直到下一代画面到达。

更稳妥的改进是同时记录当前画面控件身份；控件变化时立即重新挂载最新临时内存槽位，而不是只比较画面代数。

### 出厂页面修改能力

仓库已具备固件代码修改、设备端程序注入、校验重算和远程安装能力，不局限于图片。隐藏或重写功率、天气等出厂页面仍需要先定位页面列表、旋钮导航和页面数据回调。优先修改导航列表以隐藏页面，不直接删除仍被引用的资源。

### 数据口径必须写在标题中

“累计消耗”“近 30 天”和“单日峰值”是不同统计范围。小屏面板应使用用户熟悉的单位，并在标题中明确统计窗口，避免正确数据被误认为不一致。

## 7. 停止条件

遇到以下任一情况时，停止对应高风险步骤：

- 型号或固件版本无法精确确认；
- 原厂固件文件指纹不匹配；
- 构建报告中的修改范围超出预期；
- 固件下载地址回读不是 AP01 固件文件头；
- 仅下载验证失败；
- 用户尚未在安装前明确授权；
- 安装期间设备身份发生冲突；
- 安装后没有设备重启和重新上线的完整证据。
